# GLOBAL_AGENT_CONTRACT

## 1. Purpose

This document defines **mandatory and global rules** for all system agents (Orchestrator, Scientist, Expert, Trainer, Tester, Auditor, or any other).  
Its purpose is to ensure **scientific consistency**, **auditability**, **reproducibility**, and **technical standardization** across all agent outputs, code artifacts, and logs.

This contract is the **single source of truth**.  
If any agent-specific instruction conflicts with this document, **this contract takes precedence**.

---

## 2. Language and Tone

All outputs must adhere to an **academic, formal, and impersonal tone**.

Mandatory rules:
- No emojis of any kind.
- No slang, colloquial expressions, or informal phrasing.
- No motivational or conversational fillers.

Preferred vocabulary includes, but is not limited to:
- "Evidence"
- "Assumption"
- "Hypothesis"
- "Limitation"
- "Methodology"
- "Result"
- "Uncertainty"
- "Reproducibility"
- "Acceptance criteria"

---

## 3. Logging Standard (Mandatory)

All agents must use **a single, shared logging interface**.

### 3.1 Logger Ownership

- Agents **must not** implement or instantiate ad-hoc loggers.
- All logging must go through a centralized logger (logger.py that already exist in the project)
- Summarys, and results should be saved in markdown always with academic layout (no adjectives)

Logs **must not**:
- Contain emojis
- Contain emotional or subjective language
- Contain vague messages (e.g. "something went wrong")

---

## 4. Code Architecture Requirements

### 4.1 Class-Based Design (Mandatory)

All code artifacts must follow a **class-based architecture**.

Forbidden patterns:
- Core logic implemented as free functions
- Script-style pipelines without explicit object boundaries

Mandatory patterns:
- Explicit component classes (e.g. `DataPreprocessor`, `Trainer`, `Evaluator`, `Predictor`)
- Clear separation of responsibilities
- Deterministic constructors and explicit dependencies

---

## 5. Commenting Policy

- All comments, in any form, are forbiden.

---

## 6. Output Structure (Mandatory)

All textual outputs produced by agents must follow this structure:

1. **Objective**
2. **Assumptions**
3. **Inputs Consumed**
4. **Methodology**
5. **Findings (Evidence-Based)**
6. **Limitations and Risks**
7. **Actions and Recommendations**
8. **Artifacts Produced**
9. **Acceptance Criteria**
10. **Gate Decision** (PASS / FAIL, with justification)

No sections may be omitted.

---

## 7. Evidence and Claims

- All claims must be supported by:
  - Computed results
  - Empirical observations
  - Explicit assumptions
- Speculative language must be explicitly labeled as such.
- Assertions without traceable evidence are forbidden.

---

## 8. Reproducibility and Provenance

All agents must:
- Log all non-deterministic factors
- Track configuration, parameters, and versions
- Ensure that outputs can be reproduced given the same inputs

---

## 9. Validation and Gating

- The Orchestrator is responsible for enforcing compliance with this contract.
- Any violation (e.g. emojis, informal tone, missing sections, non-compliant logging) results in an automatic **FAIL** gate.
- Non-compliant outputs must not propagate downstream.

---

## 10. Enforcement

Compliance with this contract is **mandatory**.

Any agent that does not adhere to these rules is considered **invalid**, and its outputs must be rejected without exception.

## 11. Line Break and Formatting Discipline

All agents must **avoid unnecessary line breaks** in textual outputs.

Mandatory rules:

- Text must be written in **continuous paragraphs**, prioritizing readability and structural compactness.
- Line breaks are **forbidden** unless one of the following applies:
  - Section or subsection boundaries defined by this contract
  - Markdown-required structures (lists, code blocks, tables)
  
Artificial wrapping, stylistic line breaking, or short-line formatting is not permitted.

This rule exists to:
- Improve machine parsing
- Reduce formatting noise in audits
- Ensure consistent rendering across environments

Any violation of this formatting discipline constitutes a **contract breach**.


## 12. Markdown Artifact Consolidation

All Markdown files generated by any agent **must be stored in a single, shared directory**.

Mandatory rules:

- A unique, predefined directory must be used for **all** generated Markdown artifacts.
- Agents are **forbidden** from creating additional directories or subdirectories for Markdown outputs.
- Filenames must be:
  - Deterministic
  - Descriptive
  - Free of timestamps unless explicitly required for audit versioning
- All Markdown artifacts must be:
  - Referenced explicitly in logs and reports
  - Traceable to the experiment or agent that produced them

This rule exists to:
- Simplify audit and review workflows
- Prevent documentation fragmentation
- Ensure deterministic discovery of textual artifacts

Any Markdown file generated outside the designated directory constitutes a **contract violation** and results in an automatic **FAIL** gate.

## 13. Progress Reporting for Long-Running Loops (tqdm Mandatory)

All long-running loops must provide deterministic, standardized progress reporting via `tqdm`.

Mandatory rules:

- Any loop expected to take more than a trivial duration must use `tqdm`, including but not limited to:
  - dataset preprocessing over many files/samples
  - training epoch loops
  - evaluation loops
  - inference loops over batches
  - cross-validation or multi-seed experiment loops

- `tqdm` usage must be:
  - minimal and non-intrusive
  - compatible with deterministic execution (no side effects on RNG state)
  - configured to avoid excessive console noise (stable update frequency)

- Progress reporting must not replace structured logging.
  - `tqdm` is for operator visibility.
  - the shared logger remains mandatory for audit-grade event recording.

Any long-running loop lacking `tqdm` progress reporting constitutes a contract violation and results in an automatic **FAIL** gate.


## 14. Variable Naming Discipline (Mandatory)

The use of abbreviated or shortened variable names is strictly forbidden.

Mandatory rules:

- All variables, attributes, parameters, and configuration fields must use **fully spelled, descriptive names**.
- Abbreviations, contractions, or single-letter identifiers are not permitted, including but not limited to:
  - loop indices beyond trivial mathematical notation in local scopes
  - shortened technical terms (e.g., `lr`, `bs`, `cfg`, `tmp`, `opt`, `idx`)
- Variable names must clearly convey semantic meaning and role within the system.

Permitted exceptions (narrow and explicit):

- Standard mathematical symbols in self-contained mathematical contexts (e.g., `x`, `y` in formula translation layers only)
- Library-mandated interfaces where renaming is impossible (these must be isolated and justified)

This rule exists to:
- Improve readability and maintainability
- Reduce cognitive load during audits
- Prevent ambiguity in scientific interpretation

Any use of abbreviated variable names outside the permitted exceptions constitutes a **contract violation** and results in an automatic **FAIL** gate.
